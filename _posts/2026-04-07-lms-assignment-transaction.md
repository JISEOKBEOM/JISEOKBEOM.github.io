LMS 과제 삭제 기능에서 트랜잭션으로 데이터 정합성 문제 해결하기

1. 문제 상황

LMS 시스템에서 과제 관리 기능은 교수의 과제 생성부터 학생 제출, 평가까지 이어지는 핵심 기능이다.
하지만 과제 삭제 기능을 구현하는 과정에서 다음과 같은 문제가 발생했다.

과제를 삭제했을 때 제출 데이터만 삭제되고, 과제 데이터는 남아있는 현상

이 문제는 단순 기능 오류가 아니라 다음과 같은 서비스 문제로 이어질 수 있었다.

이미 삭제된 과제가 사용자에게 노출됨
제출 상태가 잘못 표시됨
이후 평가 및 성적 처리 과정에서 오류 가능성이 발생함

코드 구조를 확인해보면, 과제 삭제와 제출 데이터 삭제가 각각 독립적으로 실행될 수 있는 형태였다.

public int deleteProfessorAssignment(Connection con, Long assignmentId, Long professorId) throws SQLException {
    String query = QueryUtil.getQuery("professorAssignment.delete");

    try (PreparedStatement pstmt = con.prepareStatement(query)) {
        pstmt.setLong(1, assignmentId);
        pstmt.setLong(2, professorId);

        return pstmt.executeUpdate();
    }
}

public int deleteSubmissionsByAssignment(Connection con, Long assignmentId) throws SQLException {
    String query = QueryUtil.getQuery("professorAssignment.deleteSubmissionsByAssignment");

    try (PreparedStatement pstmt = con.prepareStatement(query)) {
        pstmt.setLong(1, assignmentId);
        return pstmt.executeUpdate();
    }
}

즉, 여러 작업이 하나의 단위로 묶여 있지 않기 때문에 일부 쿼리만 성공하는 경우 데이터 정합성이 깨질 수 있는 구조였다.

2. 해결 목표

이 문제를 해결하기 위해 다음과 같은 목표를 설정했다.

여러 삭제 작업을 하나의 단위로 묶는 구조 설계
삭제 과정에서 데이터 정합성 유지
오류 발생 시 전체 작업이 rollback 되는 구조 확보

핵심은 단순 삭제 기능 구현이 아니라, 연관 데이터가 항상 일관된 상태로 유지되도록 만드는 것이었다.

3. 왜 트랜잭션을 선택했는가

기존 구조에서는 다음과 같은 한계가 있었다.

DAO 단위로 쿼리가 각각 실행됨
일부 쿼리만 성공할 경우 데이터 불일치 발생
조건문이나 예외 처리만으로는 모든 케이스를 안정적으로 보장하기 어려움

따라서 필요한 것은 다음과 같은 구조였다.

모든 작업이 성공하거나, 전부 실패해야 하는 구조

이 조건을 가장 명확하게 보장할 수 있는 방법이 트랜잭션이라고 판단했다.

단순 순차 삭제 방식은 일부 실패 시 데이터 불일치가 발생할 수 있으며, 조건문 기반 예외 처리는 모든 케이스를 보장하기 어렵다. 따라서 모든 작업을 하나의 단위로 묶어 일관성을 보장하는 트랜잭션 방식을 선택했다.

4. 해결 과정
4-1. 기존 구조의 문제

기존에는 과제 삭제와 제출 데이터 삭제가 각각 독립적으로 실행될 수 있었다.

과제 삭제 / 제출 삭제
→ 각각 개별 실행
→ 일부 성공 가능
→ 데이터 정합성 깨짐

이 구조에서는 제출 데이터 삭제는 성공했지만 과제 삭제가 실패하는 경우가 발생할 수 있다. 그 결과, 사용자에게 잘못된 과제 상태가 노출될 가능성이 있었다.

4-2. 개선 방향

이 문제를 해결하기 위해 Service 계층에서 트랜잭션을 관리하도록 구조를 변경했다.

즉, Controller는 요청을 전달하고, Service는 트랜잭션을 관리하며, DAO는 실제 데이터 조작을 수행하도록 역할을 분리했다.

Client → Controller → Service(트랜잭션 관리) → DAO → DB
4-3. 핵심 변경 내용

기존에는 DAO 메서드가 독립적으로 실행되는 구조였다면, 개선 후에는 Service 계층에서 하나의 Connection을 공유하여 여러 DAO 작업을 동일한 트랜잭션 범위에서 실행하도록 변경했다.

구조적으로 보면 다음과 같은 흐름이다.

교수 요청 (과제 삭제)
→ Controller
→ Service에서 트랜잭션 시작
→ 제출 데이터 삭제
→ 과제 데이터 삭제
→ 성공 시 Commit
→ 실패 시 Rollback
→ 결과 반환

즉, 삭제 작업 중 하나라도 실패하면 전체 작업을 취소하도록 설계한 것이다.

5. Before / After
Before
DAO 독립 실행
→ 일부 성공 가능
→ 데이터 불일치 발생
After
Service 트랜잭션 관리
→ 전체 성공 / 전체 실패
→ 데이터 정합성 유지

이 차이는 단순 구현 방식의 차이가 아니라, 시스템이 데이터를 다루는 방식 자체를 바꾼 구조적 개선이라고 볼 수 있다.

6. 결과

트랜잭션 도입을 통해 다음과 같은 개선이 이루어졌다.

데이터 정합성 확보
부분 실패로 인한 오류 제거
삭제 로직 안정성 향상
오류 발생 시 안전한 복구 가능

특히 중요한 점은, 단순히 특정 오류를 막은 것이 아니라 데이터 불일치 자체가 발생하지 않도록 구조적으로 개선했다는 점이다.

7. 인사이트

이번 경험을 통해 얻은 핵심 인사이트는 다음과 같다.

데이터 정합성 문제는 기능이 아니라 구조 문제다.

처음에는 단순 삭제 기능 오류처럼 보였지만, 실제로는 여러 데이터 작업이 하나의 단위로 묶여 있지 않았기 때문에 발생한 문제였다.

즉, 쿼리 하나를 수정하는 것보다 더 중요한 것은
여러 작업을 어떤 흐름과 단위로 처리할 것인가를 설계하는 것이었다.

또한 트랜잭션은 단순한 DB 기능이 아니라, 서비스 안정성을 보장하는 핵심 설계 요소라는 점을 체감할 수 있었다.

8. 마무리

과제 삭제 기능은 겉보기에는 단순 CRUD처럼 보이지만, 실제로는 과제 데이터와 제출 데이터가 연결된 구조를 가진 기능이다.

이번 경험을 통해 기능 구현 자체보다 더 중요한 것은,
연결된 데이터가 항상 일관된 상태를 유지하도록 설계하는 것이라는 점을 배웠다.

앞으로도 단순히 동작하는 기능을 만드는 것을 넘어,
문제가 발생하지 않는 구조를 설계하는 개발자로 성장하고자 한다.
