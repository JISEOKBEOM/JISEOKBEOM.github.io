# 모듈 2 회고: 스터딩 밸리에서 2D 게임형 LMS를 설계하며 배운 것

모듈 2에서는 **StudyingValley**라는 초등학교 교육용 몰입형 LMS 프로젝트를 진행했다.

스터딩 밸리는 기존 LMS처럼 강의 목록과 버튼을 중심으로 학습을 소비하는 서비스가 아니라, 학생들이 학습을 하나의 탐험처럼 경험할 수 있도록 만든 **2D 게임형 학습 플랫폼**이다.

이번 모듈에서 내가 집중한 부분은 단순히 기능을 구현하는 것이 아니라, 요구사항을 API로 풀어내고 이후 기능이 늘어나도 유지될 수 있는 백엔드 구조를 고민하는 것이었다.

## 프로젝트 문제 정의

기존 LMS는 대부분 정적인 목록, 버튼, 페이지 이동 중심으로 구성되어 있다.

사용자는 강의를 선택하고, 콘텐츠를 보고, 다음 단계로 넘어간다. 구조는 단순하지만 학생 입장에서는 학습이 능동적인 경험이라기보다 수동적인 콘텐츠 소비에 가깝게 느껴질 수 있다.

스터딩 밸리에서 해결하고자 한 문제는 다음과 같았다.

- 기존 LMS는 단순 목록과 버튼 중심 구조이다.
- 학습이 수동적인 콘텐츠 소비 형태로 흐르기 쉽다.
- 흥미 부족으로 참여도와 지속성이 떨어질 수 있다.
- 학습 과정에서 몰입 경험이 부족하다.

그래서 프로젝트 목표는 명확했다.

> 학습을 "해야 하는 것"이 아니라 "탐험하고 싶은 것"으로 바꾸는 2D 게임형 LMS를 만든다.

## 서비스 방향

스터딩 밸리는 초등학생을 대상으로 하는 교육용 서비스이기 때문에, 사용자가 처음 서비스를 접했을 때 어렵거나 딱딱하게 느끼지 않는 것이 중요했다.

백엔드 입장에서도 이 방향은 단순히 화면을 예쁘게 만드는 문제가 아니었다. 프론트엔드에서 게임형 경험을 만들 수 있으려면, 백엔드는 화면에 필요한 데이터를 안정적으로 제공하고, 기능이 추가되어도 API 구조가 크게 흔들리지 않도록 설계되어야 했다.

즉, 모듈 2의 핵심은 다음 두 가지였다.

- 요구사항을 실제 API로 설계하고 구현하기
- 기능 확장 시에도 유지 가능한 구조 만들기

## 내가 집중한 백엔드 과제

모듈 2에서 내가 가장 크게 고민한 부분은 **API 설계와 관심사 분리**였다.

처음에는 요구사항을 보면 화면 단위로 API를 만들고 싶어진다. 예를 들어 특정 화면에서 필요한 데이터가 있으면 그 화면 전용 API를 만들고, 또 다른 화면이 생기면 별도의 API를 추가하는 방식이다.

하지만 이렇게 접근하면 기능이 늘어날수록 API가 화면에 강하게 묶인다. 화면이 조금만 바뀌어도 백엔드 API가 같이 흔들리고, 비슷한 로직이 여러 곳에 반복될 가능성이 커진다.

그래서 다음 기준을 두고 설계하려고 했다.

- 화면이 아니라 도메인과 리소스를 기준으로 API를 나눈다.
- 요청과 응답 구조를 최대한 일관되게 유지한다.
- 반복되는 로직은 서비스 내부에 흩어두지 않고 분리한다.
- 컨트롤러는 요청과 응답 처리에 집중한다.
- 실제 비즈니스 흐름은 서비스 계층에서 처리한다.
- 데이터 접근은 Repository를 통해 분리한다.

## API 설계에서 배운 점

API를 설계할 때 가장 어려웠던 점은 요구사항을 그대로 코드로 옮기는 것이 아니라, 요구사항 뒤에 있는 실제 사용 흐름을 이해해야 한다는 점이었다.

예를 들어 사용자는 단순히 버튼을 누르는 것이지만, 백엔드에서는 그 버튼이 어떤 데이터를 필요로 하는지, 어떤 상태 변경을 발생시키는지, 이후 어떤 기능과 연결될 수 있는지를 고려해야 한다.

모듈 2를 진행하면서 API를 볼 때 다음 질문을 계속 하게 되었다.

- 이 API는 어떤 사용자 행동을 처리하는가?
- 화면이 바뀌어도 유지될 수 있는 구조인가?
- 응답 데이터가 프론트엔드에서 바로 사용하기 쉬운 형태인가?
- 같은 로직이 다른 API에서 반복될 가능성은 없는가?
- 나중에 기능이 추가되면 어디를 확장해야 하는가?

이 질문들은 이후 모듈 3, 모듈 4에서 결제, 장바구니, 크레딧, 구독권 기능을 설계할 때도 계속 이어졌다.

## 관심사 분리

모듈 2에서 또 하나 중요하게 본 부분은 관심사 분리였다.

처음 기능을 구현할 때는 컨트롤러 안에서 요청을 받고, 데이터를 조회하고, 응답까지 만들어도 기능은 동작한다. 하지만 그렇게 만들면 코드가 빠르게 복잡해지고, 기능 변경이 생겼을 때 수정 범위가 커진다.

그래서 각 계층의 책임을 나누는 방향으로 구조를 잡았다.

- Controller: HTTP 요청을 받고 응답을 반환한다.
- Service: 기능 흐름과 비즈니스 로직을 조정한다.
- Repository: 데이터 저장소와의 연결을 담당한다.
- DTO/Response: 프론트엔드에 전달할 응답 형태를 정리한다.

이 구조를 통해 화면 요구사항이 바뀌더라도 전체 코드를 크게 흔들지 않고 필요한 부분만 수정할 수 있는 기반을 만들 수 있었다.

## 발생한 문제와 해결 방향

모듈 2에서 마주한 문제는 크게 세 가지였다.

첫 번째는 요구사항이 화면 중심으로 정리되어 있어, 이를 API 중심으로 다시 해석해야 했다는 점이다.

화면에서는 하나의 동작처럼 보이지만 백엔드에서는 조회, 검증, 저장, 응답 생성 등 여러 단계로 나뉜다. 그래서 단순히 화면 이름을 API 이름으로 옮기는 것이 아니라, 실제 데이터 흐름을 기준으로 API를 정리하려고 했다.

두 번째는 반복 로직 문제였다.

비슷한 조회나 응답 생성 로직이 여러 기능에서 반복될 수 있었기 때문에, 공통으로 사용할 수 있는 부분은 서비스 계층에서 분리하고 재사용 가능한 구조로 만들려고 했다.

세 번째는 확장성 문제였다.

모듈 2는 단순히 현재 기능만 끝내는 단계가 아니라, 이후 모듈에서 더 복잡한 기능을 붙이기 위한 기반이기도 했다. 그래서 기능이 추가되어도 API 구조가 쉽게 무너지지 않도록 도메인 기준으로 나누는 것을 중요하게 봤다.

## 협업에서 배운 점

이번 프로젝트를 하면서 백엔드는 단순히 데이터를 저장하고 꺼내주는 역할이 아니라는 것을 더 크게 느꼈다.

프론트엔드가 화면을 만들기 위해서는 어떤 데이터가 필요한지 명확해야 하고, 백엔드는 그 데이터를 어떤 형태로 전달할지 약속해야 한다. 이 약속이 흔들리면 프론트엔드 작업도 같이 흔들린다.

그래서 요구사항, 화면 흐름, API 명세를 계속 맞춰보는 과정이 중요했다.

특히 모듈 2에서는 다음을 배웠다.

- 요구사항은 바로 코드가 아니라 API 계약으로 먼저 정리되어야 한다.
- 프론트엔드와 백엔드는 응답 데이터 구조를 명확히 합의해야 한다.
- 기능 구현보다 먼저 데이터 흐름을 이해하는 것이 중요하다.
- 화면 중심 설계와 도메인 중심 설계의 차이를 고민해야 한다.

## 이후 모듈로 이어진 점

모듈 2에서 고민한 API 설계와 관심사 분리는 이후 모듈에서 더 중요해졌다.

모듈 3에서는 장바구니, 결제, 크레딧 기능을 구현하면서 사용자 행동이 실제 결제 흐름으로 이어지는 구조를 다뤘고, 모듈 4에서는 토스 결제 연동, 구독권, 멱등성, 모니터링처럼 운영 안정성과 연결되는 기능까지 확장했다.

이때 모듈 2에서 배운 점이 기반이 되었다.

- 화면 요구사항을 API로 바꾸는 방법
- 기능별 책임을 분리하는 방법
- 프론트엔드와 맞춰야 하는 응답 구조의 중요성
- 나중에 확장될 가능성을 고려한 설계 방식

모듈 2는 기능의 규모만 보면 이후 모듈보다 단순했지만, 백엔드 설계 관점에서는 가장 기본이 되는 사고방식을 익힌 단계였다.

## 마무리

스터딩 밸리 프로젝트를 통해 백엔드는 단순히 API를 만드는 역할이 아니라, 서비스의 흐름을 안정적으로 지탱하는 구조를 설계하는 역할이라는 것을 배웠다.

특히 모듈 2에서는 "기능이 동작한다"에서 끝나는 것이 아니라, "나중에 기능이 늘어나도 유지될 수 있는가"를 고민하기 시작했다.

이 경험은 이후 결제, 크레딧, 구독권처럼 더 민감하고 복잡한 기능을 개발할 때 중요한 기준이 되었다.

모듈 2는 나에게 백엔드 개발의 출발점이었다. 화면 뒤에서 데이터가 어떻게 흐르고, API가 어떻게 계약이 되며, 구조가 어떻게 다음 기능의 기반이 되는지를 처음으로 깊게 고민한 단계였다.
